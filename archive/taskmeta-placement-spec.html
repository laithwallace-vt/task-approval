What is a zone?
Formal Definition

A zone is any direct child of the task body that renders as a visually bounded, top-level content block — evaluated once from the form definition at render time, not from runtime state.

🔑
The key word is "top-level"
Nesting does not create new zones. Children inside a zone belong to their parent. The zone count is stable — it does not change as the user interacts with the page.

The rule
Header placement
≥ 3 zones
Body is dense. User needs all vertical space for content. Assignee + due date persist in the header bar as ambient context.
Body placement
≤ 2 zones
Body is sparse. Task is decision-focused. Assignee + due date sit below the title as reading context before the user acts.
⚡
Urgency override — always supersedes zone count
If due_date ≤ now + 24h OR due_date < now, placement is always "header" regardless of zone count. Reason: an overdue or imminent date needs to remain visible while the user works through content. Rendered with a warning colour treatment.

Maryna's questions — ruled and rationale
📋
Does a collapsed accordion count as one zone, or the zones inside it?
Ruling One zone, always. Collapsed or expanded state is irrelevant.
Why Open/closed is runtime state, not form definition state. If we counted zones inside an accordion, the total would change as the user interacts — meaning the layout could shift mid-session as someone opens an accordion. That's a worse UX than any edge case from this rule.
Exception: if the accordion is nested inside another card (e.g. "Request Details" inside vendor card), it counts as zero — it belongs to its parent zone.
🔑
Does a single key-value widget count as a zone?
Ruling Only if it's a standalone top-level container.
Test Does it have its own card border/visual container at the top level of the task body? Yes → 1 zone. No → belongs to parent zone, count 0.
A key-value block embedded inside a vendor card is part of that card. A budget summary rendered as its own card-level block is a distinct zone.
📑
Is an offer tab group one zone or N zones based on how many offers exist?
Ruling One zone. Tab count is irrelevant.
Why The tab group is the container; individual tabs are its contents. Offer count is runtime data — it changes as users add/remove offers. Zone count must be stable at form definition time.
Same principle as the accordion: the container counts once, regardless of what's inside it or how many variants exist at runtime.
⚠️
Does the reason/textarea card count — it appears in every task type?
open question Design decision needed before engineering builds
Option A Count it. Any task with 2 content zones + a reason card has 3 zones total → header. Rationale: if there are 2 substantive zones AND a required action input, the body is genuinely busy.
Option B Exclude it from zone count as structural chrome (like the title block), since it's always present and therefore not meaningful density signal.
Recommendation: Option A. The reason card adds real visual height — a task with 2 cards + textarea is genuinely dense. Maryna to confirm before this goes to engineering.
Conditional / hidden zones
⚠️
Only count zones that are rendered at task open time
Conditional zones that are hidden at task load (e.g. a risk alert that only appears when budget is exceeded) do not count toward the zone total, even if they may appear later. Zone count is evaluated once on task mount from the initial form definition render. If a conditional zone becomes visible after user interaction, the placement does not recalculate — that would cause layout shift.
Pseudocode — evaluatePlacement()
function evaluatePlacement(formDefinition, taskMeta): // 1. Urgency override — always checked first if taskMeta.dueDate ≤ now() + 24h: return { placement: "header", reason: "urgency-override", urgent: true } // 2. Count top-level zones from form definition zones = formDefinition.body .directChildren() .filter(el → isQualifyingZone(el) && isRenderedOnMount(el)) zoneCount = zones.length // 3. Apply threshold rule if zoneCount >= 3: return { placement: "header", reason: "dense", zoneCount } else: return { placement: "body", reason: "sparse", zoneCount } function isQualifyingZone(element): return element.type in [ "card", "accordion", "tab-group", "form-section", "standalone-widget", "textarea-card" ] function isRenderedOnMount(element): // Conditional zones not yet visible = excluded return element.condition == null || evaluateCondition(element.condition) == true
TaskMeta component API
// Component receives placement as a derived prop — never stored in state type TaskMetaPlacement = "header" | "body" interface TaskMetaProps { assignee: User dueDate: Date placement: TaskMetaPlacement // computed externally, never self-derived urgent?: boolean // true when urgency override applies editable?: boolean // default true — shows pencil/edit affordance } // Usage — parent computes placement, passes it down const { placement, urgent } = evaluatePlacement(formDefinition, taskMeta) // In task header bar (dense tasks): { placement === "header" && <TaskMeta placement="header" urgent={urgent} .../> } // In task body, below title (sparse tasks): { placement === "body" && <TaskMeta placement="body" .../> }
Evaluation timing
Question Answer
When is placement evaluated? Once, on task mount. Not on every render.
Does it re-evaluate if the user adds a zone? No. Placement is stable for the session. Re-evaluate on next task open.
Does it re-evaluate if the due date changes? Yes — urgency override watches dueDate reactively. Placement can switch to "header" + urgent if date is edited to be imminent.
Where does the result live? Derived at render time. Never persisted to backend. Never stored in form config.
Can admins override it? Future: a forceHeaderMeta: boolean flag in task settings panel. Not in v1.
