Why this pattern over Martin's variants ▾
✓
Segmented control for placement — Sidebar/Body are mutually exclusive. A segmented control communicates this constraint visually without needing documentation. Users cannot accidentally set both.
✓
Independent toggle for notifications — Notification inclusion is genuinely independent of placement. Any widget in any position can optionally appear in notifications. The toggle correctly models this.
✓
Notification chip as status indicator — Solves the open design question from Martin's spec: the chip confirms notification state at a glance on the widget card without requiring the user to inspect the toggle.
✗
Rejected: dual toggles (Martin variant 1) — "Show in task body" + "Show in sidebar" as two independent toggles implies both can be enabled simultaneously. That's a false model. If both are off, where does the widget go? The constraint is not surfaced.
✗
Rejected: dropdown (early mockup) — A dropdown for two mutually exclusive options adds unnecessary interaction cost. Segmented controls expose both options at once with zero additional clicks.
Design call: body widgets as collapsible sub-zone within Slot 2 — body widgets expand Slot 2 (Body) without introducing a new slot. Collapsed by default so the approval question is always the primary visual focus. Sidebar omitted when all widgets are in body. Connected IntelligenceSlot model preserved

Design decisions for the task dialog ▾
✓
Body widgets in Slot 2, not a new slot — They live above the form question as a collapsible "Context" section. No new shell slot needed. This preserves the 5-slot architecture and avoids engineering new layout modes.
✓
Collapsed by default — The approval question is always the primary visual. Context is available on demand. Power users who need the data expand; casual approvers aren't distracted by it.
✓
Notification chip on collapsed card header — Resolves Martin's open question. Workflow builders can see at-a-glance which widgets are included in notifications without opening each card.
✓
Sidebar omits tab when empty — Martin's spec says omit the sidebar tab if no widgets remain for it. Use the toggle button to see this state — sidebar hides cleanly and the main content expands.
✗
Rejected: widget above the question (Martin V1) — Puts context before the question in reading order, but the question is the primary job. Reversed hierarchy increases time-to-action for the 98% Ratification archetype who just need to approve.
